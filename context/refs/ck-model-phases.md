## Practical implications

### Per-phase

- **`/ck:sketch`** — exploration matters most here. Haiku on exploration (`balanced` default) is fine *if* the kits being revised already have grounding context (research brief, DESIGN.md, prior kits). For greenfield domains, sonnet on exploration pays off.
- **`/ck:map`** — pure reasoning. Opus reasoning under `balanced` is what you want. Upgrading costs money for no gain; downgrading loses the one model worth keeping.
- **`/ck:make`** — pure execution. This is where `balanced` vs `quality` is a real tradeoff; reasoning and exploration are irrelevant to make.

### Codex-absence impact

Codex (model family used by `/ck:judge` and the `/ck:make` tier-gate) provides *independent-model* adversarial review — it catches blind-spot bugs that same-family review (Claude reviewing Claude) can miss by sharing priors.

- **Isolated impact:** codex absence affects **only** `/ck:make` (tier-gate skipped cleanly, logs a note and continues) and `/ck:judge` (won't run). All other commands are unaffected.
- **Remaining safety net:** `/ck:check` dispatches the `ck:inspector` subagent (Claude, uses `REASONING_MODEL` = opus under `balanced`). Catches class-of-bug issues well but weaker on "A's blindspot is also B's blindspot" cases.

### Picking a preset for `/ck:make`

Consider **`quality`** (opus execution) when:
- The domain has a history of subtle bugs in-repo (e.g. layout math, width accounting, concurrency).
- The spec is dense with named pitfalls that require precise math to avoid.
- No codex → the second pass is same-family, so first-pass quality matters more.

Stay on **`balanced`** (sonnet execution) when:
- The spec is explicit and the execution is mostly wiring or CRUD.
- Task diffs will be small and human-reviewable.
- You're willing to run `/ck:check` after each wave as the safety net.

### Trust-but-verify (cheaper than running opus everywhere)

1. Stay on `balanced` for `/ck:make`.
2. Run `/ck:check` after *each* make wave (not just at tier boundaries).
3. If the inspector flags a category-X bug (border math, width accounting, race condition, etc.), flip to `quality` for the next wave covering that category.
4. Flip back to `balanced` once the bug-prone wave clears.

This buys opus execution only where it pays off, keeping the rest of the work on cheaper tokens.
